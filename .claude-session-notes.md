## Bug chave corrigido: Race condition do backlight

**Sintoma**: backlight aparecia como salvo mas sem efeito físico / setting voltava a ON

**Causa**: Sequência no boot:
1. `powerOnStateGlobal` report → `setBacklight(false)` enviado
2. `backlightControl: 1` report (estado inicial do device) chega depois
3. Listener antigo chamava `_applyBacklight(true)` → sobrescrevia `backlight_enabled = true` !

**Fix**: O listener agora verifica:
```js
if (isOn && !this.getSetting('backlight_enabled')) {
  // re-enforce sem sobrescrever setting
  cluster.setBacklight(false).catch(...)
  return;
}
this._applyBacklight(isOn);
```

## Lógica de persistência do backlight (simplificada)

**Antes**: Tinha setting separado `restore_backlight_on_boot` com opção `'force_off'`
**Agora**: Se `backlight_enabled === false` → automaticamente re-enforce OFF em:
1. Boot/init (`onNodeInit`)
2. Power restore (`attr.powerOnStateGlobal` event)
3. Qualquer report de `backlightControl: 1` quando user quer OFF

**Setting `restore_backlight_on_boot`** ainda existe no compose.json mas é ignorada na lógica. O `case 'restore_backlight_on_boot': break;` no onSettings evita "Unknown setting key" warnings.

## TuyaOnOffCluster helper methods

Registado via `clusterRegistry.js` → `app.js onInit()`. Disponível antes de qualquer device init.

| Método | Atributo ZCL | Tipo |
|--------|-------------|------|
| `setBacklight(bool)` | `backlightControl` (0x5000) | uint8 |
| `setGlobalPowerOnState(str)` | `powerOnStateGlobal` (0x8002) | enum8 |
| `setChildLock(bool)` | `childLock` (0x8000) | bool |

**IMPORTANTE**: `backlightControl` é `uint8` — device reporta `1`/`0` (não `'on'`/`'off'`). Sempre usar `!!value`.

### 2. "parse result null" no power strip — ✅ CORRIGIDO
**Causa**: `registerCapability` com `TuyaOnOffCluster` nos endpoints secundários (EP2-5). O framework faz um GET automático do `onOff` ao registar a capability. O `TuyaOnOffCluster` não consegue parsear a resposta dos EPs secundários.
**Fix**: EPs secundários usam `CLUSTER.ON_OFF` (standard) + `get: null` + `report: null`. Apenas EP1 usa `TuyaOnOffCluster` (que tem os atributos Tuya extendidos).
```
[Device:...] get → onoff → read attribute (cluster: onOff, attributeId: onOff, endpoint: 2) → parsed result null
```
